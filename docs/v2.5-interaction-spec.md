## 1. Design doctrine (the laws)

- **L1. The floorplan is the interface.** Anything the chips can do, the plan does directly: tap a room to enter, tap outside to leave, swipe to the next room. Chips demote to index + breadcrumb.
- **L2. The finger owns the glass; the machine owns the camera.** Direct manipulation is 1:1 with zero transition — no easing between finger and floorplan, ever. Animation exists only for machine-made moves (zone focus, reset, snap-back, double-tap), always one choreographed camera move.
- **L3. Never trap, never strand.** Never eat a scroll the dashboard needed; never leave the plan in a state without a one-gesture way back. Clamped pan, snap-back-to-fit, tap-outside-to-exit, Escape ladder, idle return.
- **L4. One camera.** Zoom, tilt, dim, label bloom are one physical move with internal choreography (dolly leads, crane follows). Exits always faster than entrances.
- **L5. Quiet until needed.** Nothing animates at rest. Motion means information: ripple = sensor firing, pulse = you asking "where?", glow = a light actually on. Amber is spent only on attention.
- **L6. Escalating commitment.** Glance → tap (control) → hold (detail) → mode (multi-select). Each step explicit and reversible with one gesture; no gesture jumps two levels.
- **L7 (new, per F3). The engine outranks the choreography.** No camera move ships unless the frame it runs in is cheap: gestures must not re-evaluate the scene subtree, the label engine, or backdrop filters per frame. A beautiful curve at 34fps is worse than v2.4.

## 2. Motion tokens (exact values)

Defined once on `:host` (apartment-view-card.ts:77-79), replacing literals at :118, :126, :139, :162, :169, :187, :221-224, :320, :371, :414, :447, :472:

```css
:host {
  --av-dur-instant: 90ms;   /* press feedback, check pops */
  --av-dur-fast:    180ms;  /* exits, dismissals, hover reveals */
  --av-dur-med:     320ms;  /* entrances, panel arrivals, snap-back */
  --av-dur-slow:    560ms;  /* camera moves: zone focus, reset, double-tap */

  --av-ease-out:    cubic-bezier(0.22, 1, 0.36, 1);      /* the camera, fades, labels */
  --av-ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1);   /* chips, checks, FAB; NEVER the camera */
  --av-ease-in-out: cubic-bezier(0.65, 0, 0.35, 1);      /* cross-fades, exits */
  --av-ease-snap:   cubic-bezier(0.175, 0.885, 0.32, 1.12); /* rubber-band return only */
}
@media (prefers-reduced-motion: reduce) {
  :host { --av-dur-instant: 0ms; --av-dur-fast: 0ms; --av-dur-med: 0ms; --av-dur-slow: 0ms; }
}
```

Usage map: camera = `--av-dur-slow --av-ease-out`; entrances = `--av-dur-med --av-ease-out`; exits = `--av-dur-fast --av-ease-in-out`; press/pop = `instant`..`fast` with `--av-ease-spring`; rubber-band return = `320ms --av-ease-snap`. The reduced-motion block collapses most of :523-547 into duration-zeroing — **keep** the `.tilt { transform: none !important }` override (:540-543; zeroing duration doesn't remove the tilt) and the pulse/ripple `animation: none` overrides (:530-532; keyframes don't read the duration vars).

### P0-0. `pointercancel` is an abort, never a tap (F1)

**Behavior.** A browser-claimed scroll (or palm rejection, or system gesture) that cancels our pointer stream must terminate the gesture with **no outcome**. Today `pointercancel` routes into `_onWindowPointerUp` (apartment-view-card.ts:596) where `TapHoldTracker.end()` (tap-hold.ts:54-66) returns `'tap'` for any short, still press — the instant P0-3 ships `pan-y`, a dashboard scroll starting on a marker would toggle a device.

**Implementation.**
- `apartment-view-card.ts:596` → `window.addEventListener('pointercancel', this._onWindowPointerCancel)` (and the matching remove at :614).
- New `_onWindowPointerCancel = (e: PointerEvent)`: if `_activePointers.has(e.pointerId)`: delete it; `_lastMove = null`; `_pinchStartDist = 0`; `_cancelHold()`; `_tapHold.reset()` (tap-hold.ts:68-71); `_activeMarker = null`; cancel any pending single-tap timer (P0-5) and the hold pre-fire timer (P1-5).

**Tests.** Additive regression in `card-component.test.ts`: `pointerdown` on a marker → `pointercancel` → assert **no** `callService`, no `_controlled`, no focus change; same for scene-origin. Nothing else touched.

### P0-1. Un-trap the focused state: guard removal + room swipe (F2 — this is a bug fix, not polish)

**Behavior.** Today `_onMarkerPointerDown` (:836-842) registers pointers with **no** focused guard, but `_onWindowPointerUp` early-returns while focused (:1020). Result, verified: (a) tapping a marker inside a focused zone does nothing — the flagship control surface is unreachable by pointer in the flagship state; (b) the swallowed pointerup leaks the entry in `_activePointers`, so the next single touch after exiting focus reads `size === 2` (:823) → phantom pinch → zoom jump. Fix both, and make the focused state a carousel:

- Marker tap/hold while focused works exactly as at overview.
- Horizontal swipe on the scene (dx > 56px, |dx| > 2·|dy|, duration < 350ms) → focus prev/next zone ordered by zone center-x (`zone.x + zone.width/2`; ties by center-y). Camera via `_animateTransformTo` (P0-2).
- Vertical motion → nothing (browser scrolls after P0-3; until then, no-op).
- Taps → P0-5 enter/exit logic. Pan/zoom stays suppressed (`_panZoom.setEnabled(false)` at :769 already handles this — `panBy`/`wheelZoom` return unchanged transforms).

**Implementation.**
- Delete the blanket guards at :987 (`_onWindowPointerMove`) and :1020 (`_onWindowPointerUp`). Keep :809 (`_onWheel`) but see P0-3's corrected gate; keep :820 (`_onScenePointerDown`) **removed too** — scene pointerdown must run while focused so the tap tracker feeds P0-5 and the swipe recognizer (pinch-begin at :823-830 is harmless; the disabled controller no-ops zoom).
- Swipe recognition in `_onWindowPointerUp`: when focused and `outcome === 'drag'` with `_activeMarker === null`, compute dx/dy from `_tapHold` start (expose `start` coords via a small getter on `TapHoldTracker`, additive) and elapsed time; match the thresholds above → `_focusZone(neighbor)`.
- While focused with `outcome === 'tap' && _activeMarker` → `_activateEntity` runs normally (this is the bug fix).

**Tests.** The intentional assertion change of v2.5: `apartment-view-card.zone-focus.test.ts:78-82` ("free pan/zoom is suppressed while focused", wheel at :82) stays green (the disabled controller still no-ops); any test asserting pointerup is *ignored* while focused flips to asserting classification. New regression tests in the same commit: "tap marker while focused opens control surface"; "no `_activePointers` leak across focus transitions"; "horizontal swipe while focused advances zone".

### P0-2. The engine: kill gesture lag, composite markers, gate the per-frame work (draft P0-5 + P0-7 + F3 riders + F12, one commit)

**Behavior.** Per L2/L7: during a gesture the plan and markers track the pointer with zero easing at 60fps; during machine moves, scene and markers animate as one body with identical duration/easing.

**Implementation.**
- **Transition gating (both ways, F12).** `.scene` (:126), `.tilt` (:118), **and** `.marker` position/transform transitions (:221-224) default to `none`. A single `is-animating` class on `.wrapper` enables `transition: transform var(--av-dur-slow) var(--av-ease-out)` on `.scene`, `.tilt` (with its P1-1 delay), and every `.marker` + `.marker-label` simultaneously — same token, same curve, so markers never detach from their rooms mid-flight. Keep the independent `scale`/`box-shadow`/`opacity`/`color` transitions on `.marker` (press feedback etc.) unconditional.
- **`_animateTransformTo(t: ZoomTransform)` helper:** sets `is-animating` (a `@state` boolean rendered onto `.wrapper` class), writes `_transform` **synchronously** (zone-focus tests assert transforms synchronously — verified they stay green), clears the class on `transitionend` of `.scene` (plus a `--av-dur-slow + 80ms` timeout fallback for test/JSDOM environments with no transition events). Reduced-motion: durations are 0ms, `transitionend` may not fire — the timeout fallback covers it.
- **Compositor markers (draft P0-7).** marker-overlay.ts:217-222: position via `transform: translate3d(${left}px, ${top}px, 0) translate(-50%,-50%) scale(${iconScale})` with `left:0;top:0` in the style string; labels (:271) likewise via `translate3d` + existing `--label-dy`. Drop `left`/`top` from the transition list at :221.
- **F3 rider 1 — `guard()` the scene subtree.** `_renderScene()` (:1054-1067) provably does not read `_transform` (renderBaseLayer/renderLightLayer/renderEffect take no transform; the subtree moves as one unit). Wrap its call site (:1167) in Lit's `guard([this._floor, this._cardWidth, ...relevant-state-keys], () => this._renderScene())` where the state key is a cheap composite of the floor's entity states + sun + weather (reuse `_relevantStateChanged`'s id list, :636-645, as a joined state-string). Per-frame transform renders now skip base/light/effect re-evaluation entirely.
- **F3 rider 2 — freeze the label engine during gestures.** `computeMarkerViews`'s O(n²) collision cull (marker-overlay.ts:166-191) must not run per pointermove. On gesture start (`_beginGesture`, :971) snapshot the current per-entity `labelText`/`labelAnchor` decisions; while a gesture is active, `computeMarkerViews` receives a `frozenLabels?: Map<string, string|null>` param (additive, optional) and skips the cull. Labels re-resolve on gesture end.
- **F3 rider 3 — suppress backdrop-filter in motion.** While `is-animating` **or** an active pan/pinch (`is-gesturing` class, set in `_beginGesture` when movement latches, cleared on up/cancel): `.marker`, `.marker-label` swap `backdrop-filter`/`-webkit-backdrop-filter` to `none` and the fill to the opaque `var(--card-background-color)`. Invisible in motion; removes per-frame backdrop re-sampling for ~30 blurred elements.
- **Coexist with the landed reveal system (F11):** `.marker-overlay:not(.ready) .marker { transition: none }` (:192-195) already suppresses transitions pre-reveal; the `is-animating` gate composes with it (both are "off by default, on by class").

**Tests.** `marker-overlay.test.ts:161-163` asserts `style.left === '300px'` / `style.top` / `transform` contains `translate(-50%,-50%)` — update to assert the `translate3d(300px, 250px, 0)` form in the same commit; same sweep for `preview-geometry.test.ts` if it renders markers. Zone-focus transform assertions unaffected (synchronous write). New test: `is-animating` present after `_focusZone`, absent after the fallback timeout.

### P0-3. Kill the scroll trap (draft P0-1 + P0-2 merged, with F5/F6/F7/F14 riders)

**Behavior.** The card never eats a scroll it doesn't need. Phone: vertical swipes at overview or while focused scroll the dashboard; the card owns multi-touch always and single-finger pan only when free-zoomed. Desktop: plain wheel scrolls the page unless already zoomed in free-pan; Ctrl/Cmd+wheel (and trackpad pinch, which arrives as ctrl-wheel) zooms, anchored at cursor, with sane per-device feel.

**Implementation.**
- **Three-state `touch-action`** (C4): delete static `touch-action: none` (:106); bind on `.wrapper` (:1156) in `render()`: `pan-y` when `scale === 1 && !focused`; `pan-y` while focused; `none` only when free-zoomed (`scale > 1`, unfocused). Value changes only between gestures (it's render-bound to state; state changes mid-gesture don't re-latch the browser).
- **Multi-touch escape hatch (F5).** `pan-y` lets Android Chrome steal two near-vertical fingers as a scroll and `pointercancel` the pinch. Add one non-passive `touchstart` + `touchmove` listener on `.wrapper` that calls `e.preventDefault()` **only when `e.touches.length >= 2`**. Single-finger stays native; multi-touch always belongs to the card.
- **Wheel gate (F7b corrected).** `_onWheel` (:808-817): early-return (no `preventDefault`) unless `e.ctrlKey || e.metaKey || (this._transform.scale > 1 && this._focusedZone === null)`. This keeps `zone-focus.test.ts:82` green by design (focused scale is 1.5; plain wheel over a focused room scrolls the page).
- **Wheel normalization (F6).** pan-zoom.ts:9,44: replace `WHEEL_STEP` with `const dy = e.deltaMode === 1 ? deltaY * 16 : deltaY; factor = clamp(Math.exp(-dy * 0.0022), 0.8, 1.25)`. deltaMode normalization happens in the card (`_onWheel` has the event); `wheelZoom(deltaY, ...)` receives normalized pixels and applies the exp curve. A ±100 mouse notch yields ×1.246 — matches today's ×1.1..×1.25 feel. **No legacy flag** (F7a: `pan-zoom.test.ts:17-75` wheel assertions are inequalities/derived-k — they pass under exp() unchanged; verified).
- **One-shot hint.** First plain-wheel that passes through at overview: frosted mini-pill "⌘ + scroll to zoom" (reuse `.attention-pill` styling, :353-373), `--av-dur-fast` in, fades out over 1.6s, opacity-only. Module-level flag, once per session; **suppressed when hosted inside `hui-card-preview`** (F16: check `this.closest('hui-card-preview')` once).
- **Safari riders (F14b/c).** `preventDefault()` on `gesturestart` on the host (desktop Safari trackpad pinch page-zooms otherwise, alongside its ctrl-wheel events). Base image: add `draggable="false"` + `-webkit-user-drag: none; user-select: none` to `.base-image` (:128-132 / base-layer.ts) — with `touch-action: none` gone, mouse pans would otherwise trigger native image-drag ghosts.
- P0-0 already guarantees browser-claimed scrolls abort cleanly.

**Tests.** No existing wheel dispatches in `card-component.test.ts` (verified — zero). `zone-focus.test.ts:82` passes by design. Additive: gate tests (plain wheel at scale 1 = untouched transform + no preventDefault; ctrl-wheel = zoom), deltaMode-1 normalization test in `pan-zoom.test.ts`. **Must be verified on a real HA sections dashboard before release** (memory rule: never ship HA-frontend gesture changes on "can't test locally") — phone Chrome + iOS Safari + desktop, checklist in §8.

### P0-4. Clamp + rubber-band + snap-back-to-fit (draft P0-3)

**Behavior.** Free pan can never expose gutters or strand the user: pan clamps to image-cover bounds with elastic overshoot during the drag; release animates back; releasing below scale 1.06 returns to identity.

**Implementation.**
- `PanZoomController` (pan-zoom.ts:23) gains optional `viewport?: () => Viewport`. **No viewport passed = today's unclamped math** (C3 — `pan-zoom.test.ts` constructs without it, stays green byte-for-byte). With viewport: `panBy` (:36-40) and `_applyZoom` (:62-75) clamp `panX ∈ [w(1−s), 0]`, `panY ∈ [h(1−s), 0]` — the exact cover-bounds already proven in `zoomToZone` (geometry.ts:78-79). During an active drag, overshoot beyond bounds applies resistance `excess × 0.45` (position = bound + excess·0.45). New `release(): { target: ZoomTransform; overshot: boolean }` returns the clamped rest transform.
- Card: `_syncPanZoomFromConfig` (:753-760) passes `viewport: () => this._viewport()` — safe, `_viewport()` (:745-750) is pure state, no layout reads (F11 verified). On pointerup after a drag: `const { target, overshot } = this._panZoom.release()`; if overshot → `_animateTransformTo(target)` with `320ms --av-ease-snap`; else if `scale < 1.06` → `_animateTransformTo(identity)`.

**Tests.** `pan-zoom.test.ts` untouched. Additive: clamp bounds, resistance factor, `release()` both branches, sub-1.06 snap-to-identity at card level.

### P0-5. Tap rooms on the plan + double-tap zoom (draft P0-4, with F4 hit-rects + F13 cancellation)

**Behavior.** Tapping a room **is** the zoom (C1: wayfinding wins the gesture; scene taps currently fall through unused at :1028 — verified `outcome === 'tap' && !this._activeMarker` does nothing). Double-tap free-zoom coexists via a 250ms commit window on the scene only; markers never wait (their taps stay instant).

**Implementation.**
- **Detection in `_onWindowPointerUp`** (C2 — the outcome is already classified there): when `outcome === 'tap' && !this._activeMarker`:
  1. If a previous scene tap exists within 300ms/24px → **double-tap**: cancel the pending single-tap timer; `_animateTransformTo` toggling scale 1 ↔ `min(this.config.options.zoomMax, 2)` (config.ts:225 default 1.5) anchored at the tap point (reuse `_applyZoom` anchor math through the clamp).
  2. Else arm a 250ms timer → **single tap**. The 250ms is invisible: zone focus triggers a 560ms camera anyway.
- **Hit-testing (F4 — no math inversion while tilted).** The `rotateX(11deg)` under `perspective: 1300px` (:108-109, :1159) makes inverting `markerScreenPos` (geometry.ts:40-50, pure 2D affine) wrong exactly while focused. Instead render invisible per-zone hit-rects inside `.tilt` (siblings of the marker overlay, below it in stacking): `<div class="zone-hit" data-zone-index style="left/top/width/height in %">` positioned in scene percent-space, transformed with the scene, `pointer-events: auto` **only during the single-tap resolution** (they carry `pointer-events: none` normally so they never intercept markers/pan; the tap handler resolves the zone via `shadowRoot.elementsFromPoint(x, y)` at timer fire — the browser does the 3D projection, DOM order gives smallest-zone-wins by rendering smaller-area zones later). At overview (tilt = none) this same path is exact; no dual code path.
  - Inside a zone → `_focusZone(zone)` (:766). Focused + outside all zone rects → `_exitFocus()` (:772).
- **Cancellation edges (F13).** The armed single-tap timer is cancelled by: a new `pointerdown` (any), `pointercancel` (P0-0), wheel, floor switch, Escape. Exit-by-tap is blocked while `.wrapper.is-animating` is present (gate on the class, not a fixed 250ms — the camera is 560ms).

**Tests.** Gesture currently unused → additive: single-tap focuses, double-tap toggles scale, tap-outside exits, `is-animating` blocks exit, second pointerdown cancels the timer. Zone hit resolution unit tests via `elementsFromPoint` stub in `zone-focus.test.ts`.

options:
  interaction:              # new object, all optional
    wheel: modifier         # 'modifier' (default: ctrl/cmd-wheel zooms, plain wheel scrolls)

## 8. Perf & a11y guardrails (reviewer checklist)

**Perf — reject the PR if any of these regress:**
1. No `pointermove` path re-evaluates `_renderScene()` (the `guard()` must key it off transform), runs the label collision cull, or leaves `backdrop-filter` live on markers (P0-2 riders — L7).
2. All animation is transform/opacity only; positions via `translate3d`, never `left/top` transitions. No layout reads in render or gesture paths (`_viewport()` stays pure state; `getBoundingClientRect` allowed only inside discrete event handlers, :811, :996).
3. `is-animating`/`is-gesturing` classes are always cleared (transitionend + timeout fallback) — a stuck class re-enables 560ms lag on every gesture.
4. `touch-action` value never changes mid-gesture (render-bound to state only).

**Input safety:**
5. `pointercancel` can never produce a tap/hold outcome (P0-0 test is the canary).
6. No `_activePointers` leak across any state transition (focus enter/exit, floor switch, cancel).
7. Plain wheel at overview and vertical touch at scale-1/focused must reach the dashboard — verify **on a real HA sections dashboard**: phone Chrome (Android scroll + pinch-entry, F5), iOS Safari (scroll, `gesturestart` suppression, no image-drag ghosts, F14), desktop (plain-wheel pass-through, ctrl-wheel zoom, Firefox deltaMode, F6). Memory rule: never ship on "can't test locally."

**A11y:**
8. `prefers-reduced-motion`: all duration tokens 0ms; tilt removed (`transform: none !important`); pulse/ripple `animation: none`; inertia off; every state still reachable.
9. Touch targets ≥ 44px effective (`--av-icon-size` default 44, config.ts:231; hint pill, recenter chip, HUD buttons all ≥ 44px hit area).
10. Escape never closes card state underneath an open HA dialog (F10); keyboard map is host-scoped; Tab order is transform-invariant and entity-keyed (`repeat()`, F15); `focus-visible` outlines (:231-235) untouched.
11. Zone hit-rects are `aria-hidden` and `pointer-events: none` except during tap resolution — they must never appear in the tab order or intercept marker interaction.
